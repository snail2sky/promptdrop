# PromptDrop 运营 SOP（给运营者 + AI 自己看）

## 一、日常动作（每天 30 分钟）

### 09:00 检查监控
- 看 STATE.md 是否有 fail
- 看 ORDERS.jsonl 昨日订单数

### 12:00 处理订单
- 查客服微信未读消息
- 对每张付款截图：运行 `python3 ship.py --sku XX --order XX --buyer "XX" --price XX`
- 把脚本输出的发货话术复制到微信发给买家

### 21:00 数据复盘
- 更新 docs/DAILY.md（PV/UV/订单/GMV）
- 写明天的 3 个动作

## 二、突发处理

### 情况 1：买家要退款
1. 7 天内、SKU 未使用 → 全额退款
2. 7 天后 / 已使用 → 礼貌拒绝，可送一个 9.9 的 SKU 作为补偿
3. 记录到 docs/REFUNDS.md

### 情况 2：客服微信被加爆
- 准备备用号（你的小号）
- 把 promptdrop_cs 改名为别的，备用号承接

### 情况 3：cloudflared tunnel 断了
- 主进程在 background，重新启动即可
- 看 /tmp/cf_tunnel.log 找原因

## 三、收款码管理

### 不要做的事
- ❌ 不要把收款码直接放到 HTML（会被风控扫描到敏感图）
- ❌ 不要用同一个码收太多钱（个人码有限额）

### 应该做的事
- ✅ 准备 2-3 张码轮换（你 + 家人）
- ✅ 单码日上限 ¥500，超了换下一张
- ✅ 大额订单（> ¥100）走对公转账或加好友再付

## 四、自动发货脚本使用

```bash
# 买家"小明"买了 PD-001，订单号 888888
python3 ship.py --sku PD-001 --order 888888 --buyer "小明" --price 9.9
# 复制脚本输出的微信话术 → 发给买家
```

## 五、备份策略

- 代码：已在 GitHub（https://github.com/snail2sky/promptdrop）
- 订单数据：docs/ORDERS.jsonl（建议每周手动 git push 一次）
- 客服聊天记录：微信自带备份

## 六、扩量路线（从 0 到月入 1 万）

| 阶段 | 时间 | 目标 | 动作 |
|------|------|------|------|
| Phase 1 | Day 1-3 | 第一笔成交 | 推广 |
| Phase 2 | Day 4-7 | 日均 1 单 | 优化 SKU + 多渠道 |
| Phase 3 | Week 2 | 日均 5 单 | 引入分销/联盟 |
| Phase 4 | Week 3-4 | 日均 20 单 | 自动化发货 + 客服机器人 |

## 七、止损线

如果 7 天后：
- 总订单 < 5 笔 → 切换赛道（重做选品）
- 总 GMV < 100 元 → 降价 + 找分销
- 单 SKU 0 成交 → 下架，换 SKU
